# WebRTC-compatible SDP ‚Äî mediagw analysis and Elixip design

This document has two parts:

1. an **analysis** of how the Java `mediagw-b2bua` controller
   (`../mediagw-b2bua/src/java/fr/ives/`) builds a WebRTC-compatible local SDP
   on top of the Mendooze MCU JSR309 XML-RPC API ‚Äî captured as reference
   knowledge for this project;
2. the **software design** of the equivalent capability in Elixip
   (`MediaServer.Mendooze` adapter and `MediaServer.Mockup`), taking into
   account the **delegated codec/fmtp negotiation** already implemented
   (`docs/mendooze_sdp_delegation_plan.md`) ‚Äî which the Java code predates and
   does not use.

Design only ‚Äî no implementation is part of this document.

---

## Part 1 ‚Äî How the Java mediagw builds a WebRTC SDP

### 1.1 Architecture recap

The mediagw is a SIP B2BUA (`MediaGwSipServlet`) bridging a WebRTC leg and a
SIP leg through one `MediaTranscodingSession`:

```
MediaTranscodingSession           1 JSR309 MediaSession, 2 Endpoints
 ‚îú‚îÄ‚îÄ SessionAttributes inbound     endpointId, localIceInfo, remoteIceInfo,
 ‚îú‚îÄ‚îÄ SessionAttributes outbound    remoteCryptoInfo, bitrate, ip
 ‚îú‚îÄ‚îÄ CryptoInfo sessionLocalCryptoInfo   (one DTLS identity per session)
 ‚îî‚îÄ‚îÄ MediaBridge per (MediaType, MediaRole)
      ‚îú‚îÄ‚îÄ Leg inbound  ‚îÄ‚îÄ HalfLeg snd / HalfLeg rcv (rtpMap, codecsPri, h264fmtp)
      ‚îî‚îÄ‚îÄ Leg outbound ‚îÄ‚îÄ idem
```

- `Leg.rcv` = what the remote party sends us (parsed from its SDP).
- `Leg.snd` = what we advertise in the SDP we build (`buildSndRtpMap`).
- The m= line port is the value returned by `EndpointStartReceiving`.
- `MediaBridge` decides bridging vs transcoding (`needsTranscoding` =
  different selected encoders) and drives Audio/VideoTranscoder objects.

### 1.3 ICE model: ICE-lite server, complete candidates, no trickle

- The media server is an **ICE-lite** endpoint: the SDP advertises
  `a=ice-lite`, the browser (full ICE, controlling) performs the connectivity
  checks, the server merely answers them. No client-side ICE agent exists.
- Credentials are generated by the controller (`SecureRandom`, hex: 16-char
  ufrag, 44-char pwd) and **pushed** to the server per media
  (`EndpointSetLocalSTUNCredentials`) at SDP-build time.
- Candidates: one **host UDP candidate per media**, address from
  `GetMediaCandidates`, but ‚Äî documented in the code as *"Gros HACK de la mort
  qui tue"* ‚Äî `GetMediaCandidates` does not return a usable port, so the port
  is overwritten with the `EndpointStartReceiving` return
  (`IceInfo.updatePort(l.snd.port)`). The candidate line is emitted as
  `foundation=<f>, component=1, UDP, priority=(50522114 ‚àí serverPriority),
  type host`.
- When rtcp-mux is **not** negotiated, a second candidate (component 2,
  port+1, priority+1) is synthesized for RTCP.
- Candidates are complete in the SDP (no trickle). Remote candidates from the
  peer's SDP are *not* forwarded to the server (ICE-lite learns the peer from
  the connectivity checks) ‚Äî except in a p2p relay mode where the other leg's
  parsed candidates are copied verbatim into the forwarded SDP.

### 1.4 DTLS-SRTP roles

- One DTLS identity per media session (`sessionLocalCryptoInfo`), fingerprint
  obtained from the server once (`EndpointGetLocalCryptoDTLSFingerprint`).
- **Offer**: `a=setup:actpass`. **Answer**: the stored value ‚Äî set to
  **`passive`** at session creation. Combined with ICE-lite this makes the
  media server always the **DTLS server**: the browser initiates the
  handshake. This is a field-proven constraint of mendooze ‚Äî it does not
  initiate DTLS as a client.
- Fingerprint placement mirrors the peer: if the remote SDP carried a
  session-level fingerprint, ours is emitted at session level too
  (`sessionDtls` flag), otherwise per media.
- Plain-SIP legs fall back to SDES (`a=crypto`) or clear RTP; the transport
  string degrades accordingly (`RTP/SAVP`, `RTP/AVP[F]`).

### 1.5 RTCP feedback and video refresh

- Offers always propose AVPF; answers mirror the offered profile
  (`rtcpFeedback = l.rtcpFeedback`).
- Per **video** payload type, three feedback lines are emitted:
  `nack`, `ccm fir`, `goog-remb`.
- Parsing the peer's `a=rtcp-fb` sets RTP properties on the endpoint:
  `useNACK=1` when nack is offered, plus `tmmbr=1`.
- When the leg has **no** RTCP feedback (classic SIP leg) the endpoint is
  configured with `useExtFIR=1` and the server raises a *FastUpdateRequested*
  event; the servlet converts it to a SIP INFO `picture_fast_update` on that
  leg. (B2BUA-specific ‚Äî noted for the future Elixip B2BUA module,
  `docs/b2bua_module.md`.)

### 1.6 fmtp and codec negotiation (superseded by delegation)

For the record ‚Äî this is precisely the client-side logic Elixip has delegated
to the server:

- `H264Params` parses/emits `profile-level-id`, `packetization-mode`,
  `level-asymmetry-allowed`; default `42801f`, baseline-profile PTs preferred;
  answers echo the offered parameters; the selected profile is handed to the
  video transcoder as `h264.profile-level-id`.
- OPUS is advertised with `/2` channels and
  `ptime=20;maxptime=20;minptime=20;useinbandfec=0;usedtx=0`.
- T140RED carries the RFC 4103 redundancy fmtp; H263-1996 `CIF=1;QCIF=1`.
- Codec selection is per-media-type strategy (audio: wideband-first with
  transcoding allowed; text: RED first; video: minimize transcoding, copy the
  other leg's codecs first), spread across `Leg.buildSndRtpMap*` /
  `selectEncoderForLegs`. **None of this is to be ported**: in Elixip the
  accepted PT set and fmtp are authoritative from `EndpointStartReceiving`'s
  enriched return.

### 1.7 What the Java gateway deliberately does *not* do

These absences are field-tested with browsers and inform our scope:

- **No `a=group:BUNDLE`** ‚Äî `a=mid` is mirrored but no group attribute is
  emitted, so bundling is implicitly declined; each media keeps its own
  port/transport. Works with default browser bundle policy (`balanced`);
  would fail against `max-bundle`.
- **No `a=ssrc` lines** ‚Äî an ssrc/cname/msid block exists but is commented
  out ("to be investigated, not sure"); browsers accept SDPs without it.
- **No trickle ICE** on the gateway side (complete candidates in the SDP).
- **No TURN/STUN gathering** ‚Äî host candidates only (server has public
  addresses).

### 1.8 Reality check ‚Äî a captured Chrome 142 offer

A real INVITE offer captured (2026-07-17) from the IVeS web client
(Chrome 142, jain-sip.js over WSS, kamailio in front) is kept verbatim as
`test/SDP-chrome-142-offer.txt` ‚Äî it is the reference fixture for the answer
path. What it teaches, beyond the Java-era knowledge:

- **`a=group:BUNDLE 0 1` and numeric mids** (`a=mid:0`, `a=mid:1`) ‚Äî mids
  must be echoed *verbatim* (never rebuilt from the media name), and D2
  (decline BUNDLE by not emitting the group) is exercised by every Chrome
  call, not an edge case.
- **Six H264 payload types** (39/103/107/109/115/117 ‚Äî three
  profile-level-id values √ó two packetization modes) plus VP8: exactly the
  case where delegated negotiation shines ‚Äî the *server* picks which PTs it
  accepts; the client must not assume one-PT-per-codec anywhere
  (`Sdp.negotiate`'s send map already keeps all matching PTs).
- **Two telephone-event PTs, one per clock rate** (110@48000, 126@8000) ‚Üí
  gap G10.
- **A non-RTP m= section** (`m=text 60000 TCP/WSS t140`, `a=connection:new`,
  media-level `c=IN IP4 127.0.0.1`) ‚Üí gap G9; today this section makes the
  UAS answer path fail the whole call.
- **Attributes to tolerate silently** (parse must neither choke nor forward):
  `a=extmap`, `a=extmap-allow-mixed`, `a=msid`, `a=msid-semantic`, `a=ssrc`,
  `a=ssrc-group:FID`, `a=rtcp:9 IN IP4 0.0.0.0`, `a=rtcp-rsize`,
  `a=ice-options:trickle`, TCP candidates on port 9, per-candidate
  `network-id`/`network-cost`.
- Chrome's own host candidate priorities follow the RFC 8445 formula chosen
  in D5.

## Part 2 ‚Äî Design: WebRTC SDP in Elixip

**Primary target ‚Äî a WebRTC `uac_invite` scenario for elixipp.** The design
is ordered around one concrete deliverable: an elixipp scenario placing a
call **as a WebRTC UAC** ‚Äî SIP over WSS, WebRTC offer, mendooze media ‚Äî
emulating the captured web client of ¬ß1.8 against the IVeS WebRTC gateway.
That gateway is precisely the Java code analyzed in Part 1 (the captured 401
answers `Server: IVeS-WebRTC-Gateway`), so Part 1 tells us *exactly* what our
offer must contain and what its answer will look like. The UAC path (offer
building + gateway-answer processing, ¬ß2.5) is phased first; the UAS answer
path (gaps G9/G10) stays in scope but moves to a later phase.

### 2.1 Where Elixip stands today

`MediaServer.Mendooze.Conn` + `Sdp` already implement (delegation-aware):

- DTLS fingerprint (`EndpointGetLocalCryptoDTLSFingerprint("sha-256")`),
  `a=fingerprint` + `a=setup`, remote crypto push
  (`EndpointSetRemoteCryptoDTLS` / SDES), gated by the `webrtc_support`
  conn opt (`:yes` UAC / `:if_offered` UAS);
- ICE ufrag/pwd generation + `EndpointSetLocalSTUNCredentials` /
  `EndpointSetRemoteSTUNCredentials`; trickle via `EndpointAddICECandidate`;
- `a=rtcp-mux` parsing ‚Üí `EndpointSetRTPProperties`, mirrored in answers;
- delegated codec section: accepted PTs + verbatim fmtp from
  `EndpointStartReceiving` `returnVal[1]`, send-map restriction.

**Gaps against the ¬ß1.2 checklist** (a browser would reject or fail media on
most of these):

| # | Gap | Severity |
|---|---|---|
| G1 | no `a=candidate` lines emitted | blocking ‚Äî a browser needs ‚â•1 candidate per m= section |
| G2 | no `a=ice-lite` | blocking on the UAS path ‚Äî without it a browser peer waits for checks *from* us; per D7 it is emitted in answers only |
| G3 | answer `a=setup:active` (`local_crypto_spec(state, :active)`) | blocking ‚Äî mendooze won't initiate DTLS; must be `passive` |
| G4 | DTLS protocol string `RTP/SAVPF` instead of `UDP/TLS/RTP/SAVPF` | JSEP-required form in offers (answers already mirror the offer) |
| G5 | offers never propose `a=rtcp-mux` (`rtcp_mux: false`) | required by browsers (rtcp-mux is mandatory-to-offer in WebRTC) |
| G6 | no `a=rtcp-fb` lines, no `useNACK`/`tmmbr` properties | degraded video (no NACK/FIR/REMB) |
| G7 | `a=mid` neither parsed nor mirrored | browser answers/offers carry mid; must be echoed |
| G8 | `"secure" => "1"` RTP property not set | to confirm against current server (¬ß2.7 Q2) |
| G9 | UAS answers omit offered m= sections that were filtered out (unsupported type or transport) | blocking for the *UAS* path (later phase) ‚Äî RFC 3264 ¬ß6 requires one answer m= per offer m=; jain-sip/Chrome reject a shorter answer. The captured offer (¬ß1.8) *does* trigger this |
| G10 | telephone-event PT picked as *first offered* regardless of clock rate | UAS path (later phase) ‚Äî Chrome offers one PT per clock (110@48000 for opus, 126@8000 for G.711); answering PCMU with PT 110 yields a 48 kHz DTMF mismatch |

### 2.2 Ownership split (the delegation-aware principle)

The design keeps a strict two-owner model, which is the main difference from
the Java code:

- **The media server owns the codec plane**: accepted payload types and fmtp
  (delegated negotiation, unchanged). Nothing WebRTC-specific changes here ‚Äî
  H264 `profile-level-id`/`packetization-mode` for browsers comes from the
  server verbatim.
- **The client (adapter) owns the transport plane**: ICE (`ice-lite`,
  credentials, candidates), DTLS (`fingerprint`, `setup`), `rtcp-mux`, `mid`,
  and RTCP-feedback advertisement.

One interaction point: `a=rtcp-fb` lines are **per payload type**, so their
emission iterates over the *accepted* PT set (the `rtpmaps` list of the
server-driven spec, or the codec-table PTs in the legacy fallback) ‚Äî never
over a static table.

### 2.3 `MediaServer.Mendooze.Sdp` changes (pure functions)

Types:

```elixir
@type candidate :: %{
        foundation: String.t(),      # "1"
        component: 1 | 2,
        protocol: :udp,
        priority: non_neg_integer(),
        ip: String.t(),
        port: non_neg_integer(),
        type: :host
      }

# media_desc (parse side) ‚Äî new fields
        mid: String.t() | nil,          # a=mid value, to mirror
        rtcp_fb: %{optional(non_neg_integer()) => [String.t()]},
                                        # pt => ["nack", "ccm fir", ...]; pt -1 = "*"
        candidates: [String.t()]        # raw candidate values (B2BUA forwarding later)

# media_spec (build side) ‚Äî new fields
        optional(:mid) => String.t() | nil,
        optional(:candidates) => [candidate()],
        optional(:rtcp_fb) => boolean() # emit nack/ccm fir/goog-remb per video PT
```

Functions:

- `host_candidates(ip, port, rtcp_mux?) :: [candidate()]` ‚Äî one component-1
  host candidate; adds the component-2 (`port + 1`) candidate when
  `rtcp_mux?` is false. Priority per RFC 8445 ¬ß5.1.2 (type pref 126, local
  pref 65535): `(126 <<< 24) + (65535 <<< 8) + (256 - component)` ‚Äî i.e.
  `2130706431` / `2130706430` ‚Äî replacing the Java magic constant.
  The **port is the `EndpointStartReceiving` return**, not the
  `GetMediaCandidates` one: we inherit the Java workaround knowingly (the
  server call still doesn't return a per-media port; already true for the m=
  line today).
- `build_media/1` additions (both server-driven and legacy branches):
  emit `a=mid`, `a=candidate` lines, and ‚Äî when `rtcp_fb` is true ‚Äî for each
  **video** PT of the codec section: `a=rtcp-fb:<pt> nack`,
  `a=rtcp-fb:<pt> ccm fir`, `a=rtcp-fb:<pt> goog-remb` (generic
  `{"rtcp-fb", "<pt> <val>"}` attributes, same verbatim technique as the
  delegated fmtp).
- `build/1` gains an optional session flag: `ice_lite: true` ‚Üí
  `a=ice-lite` session attribute. (Session-level ufrag/pwd stay per-media as
  today; both placements are legal, media-level is what Elixip already does.)
- `protocol_for({:dtls, _, _, _})` becomes `"UDP/TLS/RTP/SAVPF"` (G4). The
  answer path is untouched ‚Äî it mirrors the offer's protocol string.
- `parse_media/3` extracts `mid`, `rtcp_fb` and raw `candidates`.

### 2.4 `MediaServer.Mendooze.Conn` changes

No new `MediaServer.Behaviour` callback and no new conn opt: everything hangs
off the existing `webrtc_support` (`:yes` / `:if_offered` / `:no`).

State: no new field needed ‚Äî candidates derive from `local_ip` +
`local_ports[media]` at spec-build time; per-media mux/mid/fb decisions
derive from the parsed offer (`descs`), which `answer_media_spec/3` already
receives.

**UAC (offer) ‚Äî `offer_media_spec/2`**, when `webrtc?(state)`:

```elixir
crypto:    {:dtls, :actpass, hash, fingerprint}   # unchanged
protocol:  (derived) "UDP/TLS/RTP/SAVPF"          # via protocol_for fix
rtcp_mux:  true                                   # G5 ‚Äî always offered
mid:       to_string(media)                       # "audio" / "video" / "text"
candidates: Sdp.host_candidates(state.local_ip, state.local_ports[media], true)
rtcp_fb:   media == :video
```

No `a=ice-lite` in offers (D7 ‚Äî browser-shaped, ¬ß2.5). Non-WebRTC offers are
strictly unchanged.

**UAS (answer) ‚Äî `answer_media_spec/3`**, when the offer is DTLS and
`webrtc_allowed?`:

```elixir
crypto:    local_crypto_spec(state, :passive)     # G3 ‚Äî was :active
protocol:  desc.protocol                          # unchanged (mirror)
rtcp_mux:  desc.rtcp_mux                          # unchanged (mirror)
mid:       desc.mid                               # G7 ‚Äî echo the offer's mid
candidates: Sdp.host_candidates(local_ip, port, desc.rtcp_mux)
rtcp_fb:   media == :video and String.ends_with?(desc.protocol, "F")
```

plus `ice_lite: true` at session level. No `a=group:BUNDLE` is emitted or
acknowledged (Java-validated scope, ¬ß1.7); each m= keeps its own port. Note
the answer `setup` fix (G3) applies to *every* DTLS answer, browser-facing or
not ‚Äî mendooze can only be the DTLS server.

**Answer completeness (G9)** ‚Äî `Sdp.parse/1` stops dropping m= sections it
does not understand: every section is returned in offer order, non-RTP or
unknown ones as `%{supported?: false, type:, protocol:, raw_fmt:}` stubs
(supported = type in audio/video/text **and** protocol in
`RTP/AVP[F] | RTP/SAVP[F] | UDP/TLS/RTP/SAVPF`). `set_remote_offer` then
builds the answer over **all** sections, in order:

- accepted ‚Üí full media spec as below;
- unsupported, filtered by `state.medias`, or per-media negotiation failure
  (`:no_common_codec`) ‚Üí **port-0 rejection** echoing the offered protocol
  and format list verbatim (`m=<type> 0 <proto> <fmt‚Ä¶>` ‚Äî the Java
  `rejectedMedias` behavior). Only when *zero* sections are accepted does the
  call fail (`ensure_media_present` unchanged, now counting accepted ones).

This also neutralizes non-RTP sections a browser client may add ‚Äî e.g. the
captured offer's `m=text 60000 TCP/WSS t140` (text-over-WebSocket, the Java
`WebSocketLeg` feature): today its non-numeric fmt yields an empty codec list
and `Sdp.negotiate` fails the **whole call**; with G9 it becomes a clean
`m=text 0 TCP/WSS t140` rejection.

**telephone-event clock matching (G10)** ‚Äî `media_desc.dtmf_pt` becomes
`dtmf_pts :: %{clock => pt}` (parsed from the rtpmap lines); `negotiate/3`
selects the PT whose clock matches the chosen audio codec's clock (8000 for
G.711/G722, 48000 for OPUS), falling back to the 8000 one. The send map and
the answer rtpmap emit only the selected PT with its true clock.

**RTP properties** (`apply_remote_media/1` path):

- `rtcp-mux` handling unchanged (already set from the remote desc).
- New: when the remote media offers/acks feedback (`protocol` ends in `F`),
  set `%{"useNACK" => "1", "tmmbr" => "1"}` ‚Äî merged into the existing
  `EndpointSetRTPProperties` call (single RPC, merged map) (G6).
- `"secure" => "1"` pending ¬ß2.7 Q2.

**Updated RPC sequences** (delta vs `mendooze_interface.md` ¬ß4.4/¬ß4.6 flows
‚Äî the doc must be updated accordingly):

```
UAC offer:  EndpointGetLocalCryptoDTLSFingerprint
            EndpointSetLocalSTUNCredentials (per media)
            EndpointStartReceiving (per media)  -> port + accepted fmtp
            GetMediaCandidates (per media)      -> ip
            [build offer: ice-lite, UDP/TLS/RTP/SAVPF, mux, mid,
             candidates(ip, recv port), setup:actpass, rtcp-fb, delegated codecs]

UAS answer: (same local setup, gated on the offer being DTLS)
            [build answer: mirror protocol/mux/mid, setup:passive,
             candidates, rtcp-fb iff offered, delegated codecs in offerer PTs]
            EndpointSetRTPProperties (rtcp-mux/useNACK/tmmbr merged)
            EndpointSetRemoteCryptoDTLS / EndpointSetRemoteSTUNCredentials
            EndpointStartSending / EndpointStartRTPTimeout
```

### 2.5 The `uac_invite` WebRTC scenario (elixipp)

**No DSL, Behaviour or CLI change is needed** ‚Äî the plumbing is already
complete end to end:

```
scenario: send_INVITE(uri, :mediaserver, webrtc: :yes, media: :audio_video)
  ‚Üí SIP.Session.Media.get_sdp_offer(ctx, :yes, medias)
    ‚Üí create_peer_connection(server, sink, webrtc_support: :yes, media: medias)
      ‚Üí Mendooze.Conn.get_local_offer          # ‚Üê where the WebRTC offer is born
send_auth_INVITE(rsp, ‚Ä¶)                       # 407 retry reuses the same conn/offer
```

The whole feature therefore reduces to the adapter work of ¬ß2.3/¬ß2.4 plus one
new scenario file.

**What the gateway needs from our offer** (known from Part 1 ‚Äî it is the
parser of ¬ß1.1): `m=`/`c=` are used directly as the RTP destination
(candidates are parsed but not used for addressing), ICE credentials are
pushed to its endpoint, `a=fingerprint` + `a=setup:actpass` select DTLS,
AVPF is mirrored, and codecs/fmtp are negotiated leg-to-leg. Everything in
the ¬ß2.4 UAC offer spec satisfies this.

**Processing the gateway's answer** (UAC side of `set_remote_answer`): the
answer will look like ¬ß1.2 ‚Äî `a=ice-lite`, host candidates, mirrored mux/mid,
**`a=setup:passive`**. Parse tolerance for these is covered by ¬ß2.3; the two
genuinely new risk points are:

- **R1 ‚Äî DTLS client role.** Remote `setup:passive` ‚áí *our* mendooze must run
  the DTLS handshake as **client** (active). `Conn` already forwards the
  remote setup (`EndpointSetRemoteCryptoDTLS`); whether the server actually
  initiates the handshake in that role is unproven (the Java deployments only
  ever face *active* browsers) ‚Üí **Q4**, blocking for this scenario, decided
  by the real-platform E2E.
- **R2 ‚Äî ICE-lite vs ICE-lite.** The gateway never sends connectivity checks
  (lite) and neither does our mendooze; connectivity then rests on the
  gateway sending RTP straight to our `m=` port and both sides latching.
  If the gateway's endpoint gates media on a completed check, mendooze must
  emit STUN binding requests (its API accepting *remote* STUN credentials
  suggests it can) ‚Üí **Q5**, same E2E decides.
- **D7** ‚Äî consequence for our own SDP: `a=ice-lite` is emitted in **UAS
  answers only** (where Elixip behaves gateway-like). UAC offers do not claim
  it: we emulate a browser-shaped offer (¬ß1.8) and cannot honor full-agent
  duties anyway; the gateway ignores the attribute in offers.

**New scenario `scenarios/uac_invite_webrtc.exs`** ‚Äî a variant of
`uac_invite.exs`:

- `config`: WSS proxy (`proxyuri` with `transport=ws[s]` ‚Äî the transport
  layer already supports it), account/domain overridable by the external
  JSON (`-c ives.json`), `mediaserver: mendooze` via the JSON header;
- `calling`: `send_INVITE("sip:<num>@#{sip_ctx.domain}", :mediaserver,
  timeout: 90, webrtc: :yes)`; the 407 branch keeps `send_auth_INVITE`
  (the captured flow authenticates twice: kamailio 407, gateway 401
  with `qop="auth"`);
- unchanged states: `call_answered` waits `{:ms_event, _, :ice_connected}`,
  then `media_play`, then BYE.

**Acceptance** is two-staged: in CI the same scenario runs against the
enriched Mockup (¬ß2.6) acting gateway-like (`setup:passive`, mirrored
mux/mid, offerer PTs); the real check runs elixipp against the dev platform
(kamailio + IVeS-WebRTC-Gateway) and settles Q4/Q5.

### 2.6 `MediaServer.Mockup` enrichment

Goal: call-flow tests (DSL scenarios, CallUAS/CallUAC) can exercise
WebRTC-shaped offers/answers **without a media server**, with the same
`webrtc_support` semantics as the real adapter. In particular the Mockup is
the **CI stand-in for the IVeS gateway** in the `uac_invite_webrtc` scenario
(¬ß2.5): fed a WebRTC offer, it must answer gateway-like ‚Äî `setup:passive`,
mirrored protocol/mux/mid, offerer PT numbering.

Today's Mockup builds one static SDP used for both offer and answer (no
ICE/DTLS attributes despite advertising `RTP/SAVPF`, answer ignores the
offer's codecs/PT numbering/protocol). Design:

1. **Reuse `MediaServer.Mendooze.Sdp`** ‚Äî it is pure (parse / build /
   negotiate / candidates) and contains nothing mendooze-RPC-specific. The
   Mockup drops its hand-rolled `build_local_sdp`/`add_codec` helpers and
   builds `media_spec`s exactly like `Conn` does, minus the RPCs. This makes
   the mock's SDP dialect converge with the real adapter's by construction.
   (A later rename to a shared `MediaServer.Sdp` namespace is possible but
   not required; not doing it avoids churn in the delegation code.)
2. **Local material (simulated)**:
   - ufrag/pwd: same `random_token` scheme as `Conn`;
   - fingerprint: formatted from 32 random bytes
     (`AA:BB:‚Ä¶`, `sha-256`) ‚Äî no real DTLS stack behind it, which is fine:
     the mock never does a handshake, it only needs to *parse and emit*
     plausible SDP;
   - candidates: `Sdp.host_candidates/3` on the mock's real UDP socket
     ip/port (so the echo loopback keeps working);
   - `ice-lite`, `UDP/TLS/RTP/SAVPF`, `rtcp-mux`, `mid`, `rtcp-fb` ‚Äî same
     rules as ¬ß2.4.
3. **Real offer/answer asymmetry** (fixes a latent Mockup bug):
   - `get_local_offer` ‚Üí WebRTC spec iff `webrtc_support == :yes`; plain
     `RTP/AVP` otherwise (today it wrongly emits `RTP/SAVPF` for
     `:if_offered`).
   - `set_remote_offer` ‚Üí `Sdp.parse` + `Sdp.negotiate` against the mock's
     codec list; answer in the **offerer's PT numbering**, mirroring
     protocol/mux/mid/direction, `setup:passive`; WebRTC path taken iff the
     offer is DTLS and `webrtc_support in [:yes, :if_offered]`
     (offer DTLS + `:no` ‚Üí `{:error, :webrtc_not_supported}`, like `Conn`).
   - `set_remote_answer` ‚Üí `Sdp.parse`, store the negotiated view (useful to
     assert in tests); keep the delayed `:ice_connected`.
4. **No delegation simulation**: the Mockup keeps the codec-table path (it
   has no `EndpointStartReceiving` to enrich). Its fmtp output therefore
   stays client-built ‚Äî acceptable, as the mock's job is call-flow shape,
   not codec fidelity. If a test ever needs delegated-shape SDP, the
   fake-server harness of `mendooze_conn_test.exs` is the right tool.
5. Unchanged: player/recorder/echo, `:ice_connected` timing simulation,
   single RTP socket.

### 2.7 Decisions and open questions

Decided (rationale in Part 1):

- **D1** ‚Äî answer `setup` is `passive`; offers stay `actpass` (G3, ¬ß1.4).
- **D2** ‚Äî no `a=group:BUNDLE` support; `mid` mirrored only (¬ß1.7). Risk
  accepted: incompatible with `max-bundle` peers; revisit if a target
  browser/app requires it.
- **D3** ‚Äî no `a=ssrc` block (¬ß1.7).
- **D4** ‚Äî complete candidates in the SDP, host-only, no trickle emission
  (receiving trickle candidates via `add_remote_candidate/2` stays supported).
- **D5** ‚Äî RFC 8445 priority formula instead of the Java constant.
- **D6** ‚Äî candidate port = `EndpointStartReceiving` port (inherited
  workaround, ¬ß1.3).
- **D7** ‚Äî `a=ice-lite` in UAS answers only; UAC offers are browser-shaped
  and do not claim it (¬ß2.5).

Open:

- **Q1** ‚Äî should `a=rtcp-fb` also be **delegated** to the server (e.g. a
  future third element of the `EndpointStartReceiving` return), like fmtp?
  Out of scope here; the client-side emission (D-checklist G6) is designed to
  be trivially removable if the server grows this. Still open.

**Resolved server-side** (branch `feat/webrtc-improvement`,
`xmlrpc_jsr309_api.md`, 2026-07-17 ‚Äî the P1‚ÄìP5 maintainer prompts landed):

- **Q2** ‚Äî the `"secure"` RTP property is now a **no-op**: calling
  `EndpointSetRemoteCryptoDTLS` (or SDES) arms SRTP implicitly. `Conn`
  correctly omits it (phase 2). ‚úÖ
- **Q3 / D6** ‚Äî `GetMediaCandidates` now returns the media's actual receive
  port (same as `EndpointStartReceiving`). D6 stays as-is (candidates use the
  receive port either way); the workaround is no longer *needed* but is
  harmless. ‚úÖ
- **Q4** ‚Äî mendooze now takes the DTLS **client** (active) role when the
  remote answers `setup:passive`, emitting the ClientHello once a send
  destination exists. `Conn` forwards the remote `setup` verbatim; no client
  change needed (phase 2). ‚úÖ
- **Q5** ‚Äî `EndpointSetRemoteSTUNCredentials` now triggers **outbound STUN
  binding requests** (ICE-controlling, aggressive nomination) toward the
  `EndpointStartSending` destination, so ICE-lite ‚Üî ICE-lite connectivity is
  established. ‚úÖ
- **P5** ‚Äî a positive `EndpointConnectedEvent` (type 7) now fires per media on
  first validated RTP + DTLS-up. The synthetic `:ice_connected` has been
  replaced by it: the poller decodes event 7 and the Conn emits a single
  connection-level `:ice_connected` (`connected_notified`). ‚úÖ

The real-platform E2E (phase 4) remains the acceptance test, but Q4/Q5 are no
longer expected blockers.
